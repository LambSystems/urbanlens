# hybrid_thermal Integration

`hybrid_thermal` is the RGB-to-thermal model package inside ThermalGen. It is not a second product or a parallel repo inside UrbanLens.

Canonical naming:

- Product name: `ThermalGen`
- Model package and folder name: `hybrid_thermal`
- Python package: `backend/app/thermal/hybrid_thermal`
- Runtime data: `backend/data/hybrid_thermal`
- Local model artifacts: `backend/models/hybrid_thermal`

Do not use `hybrid-thermal` or a root-level `HybridThermal` folder. Those were staging names. The canonical filesystem/import name is `hybrid_thermal`.

## Role In ThermalGen

`hybrid_thermal` provides thermal evidence for the agent. ThermalGen still owns:

- source retrieval
- evidence normalization
- candidate discovery
- agent investigation traces
- discard logic
- scoring and ranking
- recommendation output

The product story remains:

```text
select region -> detect hotspots -> investigate -> discard -> prioritize -> recommend
```

The model story is narrower:

```text
RGB drone image -> hybrid_thermal -> predicted thermal map -> thermal cues for hotspots
```

## Local Files

These are intentionally ignored by Git because they are local data/artifacts or generated:

- `backend/data/hybrid_thermal/Predict_Thermal/`
- `backend/data/hybrid_thermal/Test_RGB_centercrop_640x512/`
- `backend/data/hybrid_thermal/uploads/`
- `backend/models/hybrid_thermal/checkpoints/`
- root-level virtualenvs and caches

These are tracked:

- `backend/models/hybrid_thermal/config.yaml`
- `backend/models/hybrid_thermal/config.yaml.example`

Share checkpoints as a zip outside Git, for example through Google Drive. No dataset folder is required for the app workflow. After unzipping, teammates should have:

```text
backend/models/hybrid_thermal/checkpoints/best_psnr.pth
```

This avoids Git LFS and keeps the repository fast to clone.

## Fast One-Image Inference

Use the app wrapper or `POST /thermal/infer/upload`. The current product path does not require a bundled dataset.

## Preprocessing Scope

The old dataset pipeline is not part of the app flow anymore. Teammates do not need to run manifest builders, split scripts, AlphaEarth export, or batch alignment.

The runtime still preprocesses each single input image because the checkpoint expects a fixed tensor shape and normalization. That per-image step happens inside `backend/app/thermal/hybrid_thermal/runtime.py` and writes generated files to:

```text
backend/data/hybrid_thermal/Test_RGB_centercrop_640x512/
backend/data/hybrid_thermal/Predict_Thermal/
```

Those folders are runtime artifacts and stay ignored.

## App Wrapper

The app-level call is:

```python
from backend.app.thermal.generator import generate_thermal

result = generate_thermal("path/to/rgb.JPG", {"lat": 38.6270, "lng": -90.1994})
```

The current signature also supports explicit output paths and metadata:

```python
result = generate_thermal(
    "path/to/rgb.JPG",
    {"center": {"lat": 38.6270, "lng": -90.1994}, "source": "agent_tool"},
    output_path="backend/data/hybrid_thermal/Predict_Thermal/example.png",
    allow_fallback=False,
)
```

The wrapper returns:

- `thermal_image_path`
- `thermal_image_url`
- `thermal_preview_path`
- `thermal_preview_url`
- `thermal_data.min_temp_c`
- `thermal_data.max_temp_c`
- `thermal_data.mean_temp_c`
- `thermal_data.hotspot_regions`

`thermal_image_path` is the grayscale model output used by the pipeline. `thermal_preview_url` is the browser-loadable autocontrasted orange visualization generated by the backend for UI display.

If the image or checkpoint is unavailable, the wrapper returns no hotspot regions and includes an error in `thermal_data`. The app should show that thermal evidence is unavailable rather than fabricating demo hotspots.

For teammate API usage, see `docs/thermalgen_tool.md`.
